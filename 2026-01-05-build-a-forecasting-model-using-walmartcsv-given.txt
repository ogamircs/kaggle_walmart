     Walmart Sales Forecasting - Implementation Plan

     Overview

     Build a sales forecasting system for Walmart weekly sales data with 
     multiple ML models, optimize for WMAPE, and create a Gradio WebUI
     dashboard.

     Data Summary

     - File: Walmart.csv - 6,435 rows (45 stores × 143 weeks)
     - Date Range: Feb 5, 2010 - Oct 26, 2012
     - Target: Weekly_Sales (Mean: ~$1M, Range: $210K - $3.8M)
     - Features: Store, Date, Holiday_Flag, Temperature, Fuel_Price, CPI, 
     Unemployment
     - Data Quality: No missing values, no duplicates

     Project Structure

     walmart_kaggle/
     ├── Walmart.csv                 # Source data
     ├── requirements.txt            # Dependencies
     ├── src/
     │   ├── __init__.py
     │   ├── config.py              # Constants, paths, holiday dates
     │   ├── data_preprocessing.py  # Load, clean, split data
     │   ├── feature_engineering.py # All feature creation logic
     │   ├── metrics.py             # WMAPE and evaluation metrics
     │   └── models.py              # All model implementations
     ├── app/
     │   ├── __init__.py
     │   └── gradio_app.py          # Main Gradio WebUI
     ├── models/                    # Saved trained models
     └── train.py                   # Main training script

     Implementation Steps

     Step 1: Setup & Configuration (src/config.py)

     - Define holiday dates (Super Bowl, Labour Day, Thanksgiving, 
     Christmas)
     - Set paths, random seeds, validation cutoff date (Sep 1, 2012)

     Step 2: Data Preprocessing (src/data_preprocessing.py)

     - Load CSV with proper date parsing (DD-MM-YYYY format)
     - Sort by Store and Date
     - Time-based train/validation split:
       - Training: Feb 2010 - Aug 2012 (~134 weeks × 45 stores = 6,030 
     samples)
       - Validation: Sep-Oct 2012 (~9 weeks × 45 stores = 405 samples)

     Step 3: Feature Engineering (src/feature_engineering.py)

     Create comprehensive features for best WMAPE:

     | Category     | Features                                              
                                             |
     |--------------|-------------------------------------------------------
     ----------------------------------------|
     | Temporal     | year, month, week_of_year, quarter, is_month_start/end
                                             |
     | Cyclical     | month_sin/cos, week_sin/cos (capture periodicity)     
                                             |
     | Holiday      | is_super_bowl, is_thanksgiving, is_christmas, 
     is_labour_day, weeks_to_holiday, is_pre_holiday |
     | Lag          | sales_lag_1, lag_2, lag_4, lag_12, lag_52 (same week 
     last year)                               |
     | Rolling      | rolling_mean_4w/8w/12w, rolling_std_4w, ewma features 
                                             |
     | Store        | store_mean_sales, store_std, store_rank (from training
      data only)                             |
     | Economic     | temp_deviation, fuel_change, cpi_change, 
     unemployment_category                                |
     | Interactions | store × holiday, temp × season                        
                                             |

     Step 4: Metrics (src/metrics.py)

     WMAPE = SUM(|actual - predicted|) / SUM(actual) × 100
     - Weights predictions by actual sales (higher sales = more important)
     - Also implement MAE, RMSE, R² for comparison

     Step 5: Models (src/models.py)

     | Model        | Type        | Approach              | Tuning          
                    |
     |--------------|-------------|-----------------------|-----------------
     ---------------|
     | RandomForest | ML          | Global (all stores)   | Optuna          
                    |
     | XGBoost      | ML          | Global (all stores)   | Optuna          
                    |
     | LightGBM     | ML          | Global (all stores)   | Optuna          
                    |
     | SARIMA       | Time Series | Per-store (45 models) | Grid search     
                    |
     | Prophet      | Time Series | Per-store (45 models) | Default params  
                    |
     | Ensemble     | Meta        | Weighted average      | Optimize weights
      on validation |

     Hyperparameter Tuning: Use Optuna with 20 trials (quick mode), 
     optimizing WMAPE on time-series cross-validation.

     Step 6: Training Pipeline (train.py)

     1. Load and preprocess data
     2. Create features (handle lag NaNs for early weeks)
     3. Split train/validation by date
     4. Train each model with hyperparameter tuning
     5. Evaluate on validation set
     6. Save best models and results

     Step 7: Gradio WebUI (app/gradio_app.py)

     Tab 1: EDA
     - Sales distribution histogram
     - Sales over time (line chart)
     - Sales by store (bar chart)
     - Holiday vs non-holiday comparison (box plot)
     - Monthly seasonality pattern
     - Feature correlation heatmap

     Tab 2: Model Results
     - Performance comparison table (WMAPE, MAE, RMSE, R²)
     - Actual vs Predicted chart
     - WMAPE by store (bar chart)
     - Residual distribution
     - Feature importance (top 20)

     Tab 3: Predictions
     - Input form: Store, Date, Temperature, Fuel_Price, CPI, Unemployment, 
     Holiday
     - Predict button → Display predicted sales
     - Show similar historical weeks

     Key Technical Decisions

     1. Validation Split: Time-based (not random) to simulate real 
     forecasting
     2. Lag Feature Handling: Drop rows with NaN lags OR fill with store 
     mean
     3. Target Encoding: Use k-fold to prevent leakage
     4. Per-Store Models: SARIMA/Prophet trained per store; tree models are 
     global
     5. Feature Importance: Use SHAP for interpretability

     Expected WMAPE Targets

     - Baseline (mean prediction): ~15-20%
     - Good model: < 8%
     - Excellent model: < 5%
